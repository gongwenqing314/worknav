## 1. 设计概述

### 1.1 项目背景

心智障碍人士（包括智力障碍、自闭症谱系障碍等）在融入职场过程中面临诸多挑战：任务理解困难、沟通表达受限、环境适应能力弱、突发情况应对不足等。目前市场上缺乏专门针对该群体工作场景的数字化支持工具。

"工作导航"项目旨在通过移动端技术，为心智障碍员工提供可视化的任务引导、便捷的沟通辅助和即时的远程支持，同时为辅导员（支持人员）提供高效的管理后台，形成"员工-辅导员-家长-雇主"多方协作的支持生态。

### 1.2 产品定位

| 维度 | 说明 |
|------|------|
| 产品名称 | 工作导航（StepByStep） |
| 产品类型 | 移动应用 + Web 管理后台 |
| 目标用户 | 心智障碍员工、辅导员、家长、雇主 |
| 核心价值 | 降低心智障碍人士的工作难度，提升其职场适应能力与独立性 |
| 使用场景 | 工作场所（工厂、办公室、服务行业等） |

### 1.3 核心功能概述

#### 1.3.1 任务拆解与视觉化流程导航

- **辅导员端**：创建多步骤任务模板，每个步骤可附加图片、视频、语音指导
- **员工端**：接收任务后逐步执行，每完成一步自动进入下一步，支持进度可视化
- 支持任务模板复用、版本管理、定时推送

#### 1.3.2 沟通板与情绪温度计

- **沟通板**：预设常用语卡片，点击即可语音播报，支持自定义常用语
- **情绪温度计**：员工可选择情绪等级（开心/平静/不安/难过/愤怒），辅导员端实时接收
- **预警机制**：情绪持续低落或触发预设规则时，自动通知辅导员和家长

#### 1.3.3 环境适应与即时支持

- **上下班提醒**：定时推送打卡提醒、工作任务提醒
- **应急卡片**：一键求助，发送定位与紧急信息
- **远程协助**：拍照 + 圈画标注 + 录音回传，辅导员实时远程指导

#### 1.3.4 后台管理

- **任务管理**：任务模板 CRUD、任务分配、执行监控
- **员工管理**：员工档案、能力评估、工作记录
- **数据统计**：任务完成率、情绪趋势、工作时长统计
- **消息通知**：系统通知、预警推送、远程协助请求

### 1.4 设计目标与原则

| 目标 | 说明 |
|------|------|
| 易用性 | 员工端操作极简化，大按钮、高对比度、语音辅助 |
| 可靠性 | 支持离线模式，网络恢复后自动同步 |
| 实时性 | 消息推送延迟 < 3 秒，远程协助响应 < 5 秒 |
| 安全性 | 数据加密传输存储，严格的权限控制 |
| 可扩展性 | 模块化设计，支持后续功能扩展 |
| 无障碍 | 符合 WCAG 2.1 AA 级无障碍标准 |

### 1.5 术语说明

| 术语 | 说明 |
|------|------|
| 员工 | 使用本系统的心智障碍人士 |
| 辅导员 | 为员工提供工作指导和支持的专业人员 |
| 家长 | 员工的监护人或家属 |
| 雇主 | 员工所在企业的管理人员 |
| 任务模板 | 辅导员创建的可复用的多步骤任务定义 |
| 任务实例 | 分配给具体员工的任务执行记录 |
| 沟通板 | 预设常用语的交互式卡片面板 |
| 情绪温度计 | 用于员工表达当前情绪状态的功能模块 |

```
│  │  └──────┘ └───────┘  │    │  └────────┘ └────────────┘  │   │
│  └──────────────────────┘    └──────────────────────────────┘   │
├─────────────────────────────────────────────────────────────────┤
│                     业务逻辑层（Business Logic Layer）             │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                    API 网关（Nginx / Kong）                │   │
│  └──────────────────────────────────────────────────────────┘   │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────────┐   │
│  │用户服务│ │任务服务│ │消息服务│ │文件服务│ │远程协助服务│   │
│  └────────┘ └────────┘ └────────┘ └────────┘ └────────────┘   │
│  ┌────────┐ ┌────────┐ ┌──────────────────────────────────┐   │
│  │情绪服务│ │统计服务│ │      通知推送服务（WebSocket）     │   │
│  └────────┘ └────────┘ └──────────────────────────────────┘   │
├─────────────────────────────────────────────────────────────────┤
```

#### 2.2.3 离线同步策略

离线模式是本系统的核心能力之一，确保员工在无网络环境下仍可正常使用任务执行和沟通板功能。

**离线支持范围：**

| 功能 | 离线支持 | 同步策略 |
|------|----------|----------|
| 任务执行 | 支持 | 恢复网络后上传执行记录 |
| 沟通板 | 支持 | 常用语本地缓存，自定义语同步 |
| 情绪记录 | 支持 | 恢复网络后上传记录 |
| 消息通知 | 不支持 | 恢复网络后拉取未读消息 |
| 远程协助 | 不支持 | 需要网络连接 |
| 应急求助 | 降级支持 | 本地记录 + 网络恢复后重发 |

**同步机制设计：**

```
┌─────────────────────────────────────────────────────┐
│                   离线同步流程                        │
│                                                     │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐    │
│  │ 本地操作  │ ──> │ 本地队列  │ ──> │ 网络检测  │    │
│  │ 生成数据  │     │ 暂存     │     │ 连通性   │    │
│  └──────────┘     └──────────┘     └────┬─────┘    │
│                                          │          │
│                              ┌───────────┴──────┐   │
│                              │                  │   │
│                         在线 / 离线         在线      │
│                              │                  │   │
│                    ┌─────────┴──────┐    ┌────┴───┐ │
│                    │ 等待网络恢复    │    │ 立即同步│ │
│                    │ 定时重试        │    │        │ │
│                    └────────────────┘    └────────┘ │
│                              │                  │   │
│                              └───────┬──────────┘   │
│                                      │              │
│                              ┌───────┴──────┐       │
│                              │  同步引擎     │       │
│                              │  增量同步     │       │
│                              │  冲突解决     │       │
│                              └──────────────┘       │
└─────────────────────────────────────────────────────┘
```

**同步策略详细说明：**

1. **本地优先（Local-First）**：所有写操作先写入本地 SQLite 数据库
2. **操作队列**：使用 SQLite 表记录待同步操作，包含操作类型、数据内容、时间戳
3. **增量同步**：基于时间戳的增量同步，每次同步仅传输变更数据
4. **冲突解决**：
   - 任务执行记录：以服务端时间戳为准，本地记录作为补充
   - 情绪记录：合并处理，保留所有记录
   - 常用语更新：服务端优先，覆盖本地
5. **数据预加载**：登录时预加载当前任务、常用语、沟通板配置等核心数据
6. **存储空间管理**：本地缓存设置上限（建议 500MB），超出后按 LRU 策略清理

## 4. 数据库设计

### 4.1 设计原则

1. **规范性**：遵循第三范式（3NF），适当冗余以优化查询性能
2. **可扩展性**：预留扩展字段，使用 JSON 类型存储灵活数据
3. **软删除**：核心表使用 `is_deleted` 字段标记删除，保留数据可追溯
4. **审计字段**：所有表包含 `created_at`、`updated_at` 时间戳
5. **字符集**：使用 `utf8mb4` 字符集，支持 emoji 表情存储
6. **索引策略**：高频查询字段建立索引，兼顾写入性能

## 5. API 设计规范与核心接口列表

### 5.1 RESTful 设计规范

#### 5.1.1 URL 规范

| 规则 | 说明 | 示例 |
|------|------|------|
| 使用小写字母 | URL 路径全部小写 | `/api/v1/task-templates` |
| 使用连字符 | 多词用连字符连接 | `/task-instances` |
| 使用名词复数 | 资源用复数形式 | `/users`、`/tasks` |
| 层级关系 | 用路径表示嵌套关系 | `/task-templates/{id}/steps` |
| 版本控制 | URL 中包含版本号 | `/api/v1/...` |
| 查询参数 | 过滤、排序、分页用查询参数 | `?status=active&page=1&size=20` |

#### 5.1.2 HTTP 方法规范

| 方法 | 用途 | 幂等性 | 示例 |
|------|------|--------|------|
| GET | 查询资源 | 是 | `GET /api/v1/task-templates` |
| POST | 创建资源 | 否 | `POST /api/v1/task-templates` |
| PUT | 全量更新资源 | 是 | `PUT /api/v1/task-templates/1` |
| PATCH | 部分更新资源 | 否 | `PATCH /api/v1/task-instances/1/status` |
| DELETE | 删除资源 | 是 | `DELETE /api/v1/task-templates/1` |

#### 5.1.3 状态码规范

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| 200 | 成功 | GET、PUT、PATCH、DELETE 成功 |
| 201 | 已创建 | POST 创建资源成功 |
| 204 | 无内容 | DELETE 成功（无返回体） |
| 400 | 请求错误 | 参数校验失败、请求格式错误 |
| 401 | 未认证 | Token 缺失或过期 |
| 403 | 无权限 | 权限不足 |
| 404 | 未找到 | 资源不存在 |
| 409 | 冲突 | 资源冲突（如重复创建） |
| 429 | 请求过多 | 触发限流 |
| 500 | 服务器错误 | 服务端内部异常 |

### 5.3 认证机制

所有需要认证的接口，请求头需携带 JWT Token：

Token 有效期配置：

| Token 类型 | 有效期 | 说明 |
|------------|--------|------|
| Access Token | 2 小时 | 短期令牌，用于接口认证 |
| Refresh Token | 30 天 | 长期令牌，用于刷新 Access Token |

## 6. 安全设计

### 6.1 数据加密

#### 6.1.1 传输加密

| 项目 | 方案 | 说明 |
|------|------|------|
| HTTPS | TLS 1.3 | 所有 API 通信强制使用 HTTPS |
| 证书 | Let's Encrypt / 云厂商证书 | SSL 证书自动续期 |
| HSTS | 启用 | 强制 HTTPS，防止降级攻击 |
| WebSocket | WSS（TLS 加密） | WebSocket 连接使用 WSS 协议 |

#### 6.1.2 存储加密

| 数据类型 | 加密方案 | 说明 |
|----------|----------|------|
| 用户密码 | bcrypt（cost=12） | 单向哈希，不可逆 |
| 敏感个人信息 | AES-256-GCM | 手机号、身份证号等加密存储 |
| 本地数据库（SQLite） | SQLCipher | 移动端本地数据库加密 |
| Redis 敏感数据 | AES-256 | Token 黑名单等敏感缓存加密 |
| 文件存储 | 服务端加密（SSE） | 对象存储启用服务器端加密 |

#### 6.1.3 密钥管理

| 项目 | 方案 |
|------|------|
| 密钥存储 | 使用云厂商 KMS（密钥管理服务） |
| 密钥轮换 | 每 90 天自动轮换加密密钥 |
| 环境隔离 | 开发/测试/生产环境使用不同密钥 |
| 配置管理 | 密钥通过环境变量注入，不写入代码仓库 |

### 6.2 身份认证（JWT）

#### 6.2.1 JWT 结构

```
Header:  { "alg": "RS256", "typ": "JWT" }
Payload: { "sub": "用户ID", "role": "角色", "exp": "过期时间", ... }
Signature: RS256 签名
```

#### 6.2.2 认证流程

```

│  客户端  │         │  服务端  │         │  Redis  │
└───┬────┘         └───┬────┘         └───┬────┘
    │  1. 登录请求     │                  │
    │───────────────>│                  │
    │                  │  2. 验证凭据     │
    │                  │                  │
    │  3. 返回 JWT     │                  │
    │<───────────────│                  │
    │                  │                  │

    │  请求 API        │                  │
    │───────────────>│                  │
    │                  │  5. 验证签名     │
    │                  │  6. 检查黑名单   │
```

#### 6.2.3 Token 安全策略

| 策略 | 说明 |
|------|------|
| 签名算法 | RS256（非对称加密，私钥签名，公钥验证） |
| Access Token 有效期 | 2 小时 |
| Refresh Token 有效期 | 30 天 |
| Token 黑名单 | 登出时将 Token 加入 Redis 黑名单，TTL 与 Token 有效期一致 |
| Token 续期 | Access Token 过期前 15 分钟自动续期 |
| 多端登录 | 支持，但限制同一角色最多 3 个活跃设备 |
| 设备绑定 | Token 绑定设备指纹，防止跨设备使用 |

**设备管理与换机流程：**

1. **首次绑定**：辅导员创建员工账号后，在员工手机上登录APP，系统自动绑定当前设备指纹
2. **换机申请**：员工更换设备后，在新设备上打开APP，显示"请联系辅导员绑定新设备"提示
3. **辅导员授权**：辅导员在Web端"设备管理"页面查看该员工的设备列表，点击"生成临时绑定码"（6位数字，5分钟有效）
4. **员工激活**：员工在新设备上输入绑定码，系统验证通过后完成新设备绑定，旧设备自动解绑
5. **安全策略**：每个员工同时仅允许一台设备在线；绑定码使用后立即失效；辅导员可远程强制解绑设备

```javascript
// 权限中间件示例
const rbacMiddleware = (allowedRoles) => {
  return (req, res, next) => {
    const userRole = req.user.role;
    if (!allowedRoles.includes(userRole)) {
      return res.status(403).json({
        code: 403,
        message: '权限不足，无法访问该资源'
      });
    }
    next();
  };
};

// 数据级别权限控制
const dataScopeMiddleware = (req, res, next) => {
  const currentUser = req.user;
  const targetUserId = req.params.id;

  // 员工只能访问自己的数据
  if (currentUser.role === 'employee' && currentUser.id !== targetUserId) {
    return res.status(403).json({ code: 403, message: '权限不足' });
  }

  // 辅导员只能访问关联员工的数据
  if (currentUser.role === 'counselor') {
    // 检查是否为关联员工
  }

  next();
};
```

## 8. UI/UX 设计规范

### 8.1 设计理念

本系统的核心用户群体为心智障碍人士，UI/UX 设计遵循以下核心理念：

| 理念 | 说明 |
|------|------|
| **极简主义** | 界面元素最少化，减少认知负担 |
| **视觉引导** | 通过颜色、图标、动画引导操作流程 |
| **一致性** | 统一的交互模式和视觉风格，降低学习成本 |
| **容错性** | 允许撤销操作，防止误操作造成困扰 |
| **多模态** | 文字 + 图片 + 语音 + 动画多通道信息传达 |
| **渐进披露** | 信息分层展示，避免一次性呈现过多内容 |

### 8.2 色彩体系

#### 8.2.1 主色板

色彩体系采用高对比度设计，确保视觉障碍用户也能清晰辨识。

| 用途 | 色值 | 色名 | 说明 |
|------|------|------|------|
| 主色 | `#2196F3` | 蓝色 | 品牌色，用于主要按钮和导航 |
| 主色（深） | `#1565C0` | 深蓝 | 主色按下状态 |
| 主色（浅） | `#BBDEFB` | 浅蓝 | 选中背景、标签背景 |
| 辅助色 | `#4CAF50` | 绿色 | 成功、完成、正向反馈 |
| 警告色 | `#FF9800` | 橙色 | 警告、提醒 |
| 危险色 | `#F44336` | 红色 | 错误、紧急、危险操作 |
| 信息色 | `#00BCD4` | 青色 | 提示信息 |

| 情绪 | 色值 | 说明 |
|------|------|------|

### 8.3 字体规范

#### 8.3.1 字体选择

| 平台 | 字体 | 说明 |

| Android | Noto Sans CJK SC | Google 开源中文字体 |
| Web | PingFang SC, Noto Sans CJK SC, Microsoft YaHei | 优先使用苹方，回退到思源黑体、微软雅黑 |

#### 8.3.2 字号规范

| 用途 | 员工端字号 | 辅导员/Web 端字号 | 字重 |
|------|-----------|------------------|------|
| 页面标题 | 24sp | 24px | Bold (700) |
| 模块标题 | 20sp | 20px | Bold (700) |
| 卡片标题 | 18sp | 16px | SemiBold (600) |
| 正文 | 18sp | 14px | Regular (400) |
| 辅助文字 | 16sp | 12px | Regular (400) |
| 按钮文字 | 20sp | 16px | SemiBold (600) |
| 最小可读文字 | 16sp | 12px | Regular (400) |

> **注意**：员工端字号整体放大 2-4sp，确保心智障碍用户可清晰阅读。

#### 8.3.3 行高与间距

| 项目 | 员工端 | Web 端 |
|------|--------|--------|
| 行高 | 字号的 1.5-1.8 倍 | 字号的 1.5-1.6 倍 |
| 段落间距 | 16dp | 16px |

### 8.4 图标规范

#### 8.4.1 图标风格

| 属性 | 规范 |
|------|------|
| 风格 | 圆角线性图标（Round Linear） |
| 尺寸 | 员工端 48dp / Web 端 24px |
| 线宽 | 2dp / 2px |
| 圆角 | 2dp / 2px |
| 颜色 | 跟随当前文字颜色 |
| 图标库 | Material Icons / 自定义图标 |

#### 8.4.2 图标使用原则

| 原则 | 说明 |
|------|------|
| **图标 + 文字** | 所有可操作元素必须同时包含图标和文字标签 |
| **语义明确** | 优先使用用户熟悉的具象图标，避免抽象表达 |
| **一致性** | 同一功能在不同页面使用相同图标 |
| **可识别性** | 图标最小可点击区域 48x48dp（员工端 64x64dp） |
| **色彩区分** | 不同功能模块使用不同颜色图标辅助区分 |

#### 8.5.5 反馈机制

| 反馈类型 | 场景 | 实现方式 |

### 8.6 无障碍设计要点

#### 8.6.1 视觉无障碍

| 项目 | 要求 |
|------|------|
| 色彩对比度 | 所有文字与背景对比度满足 WCAG AA 标准（正文 4.5:1） |
| 不仅依赖颜色 | 不单独使用颜色传达信息，同时使用图标/文字/形状辅助 |
| 文字可缩放 | 支持系统级字体缩放（最大 200%），布局自适应 |
| 深色模式 | 支持深色模式切换，降低视觉疲劳 |
| 动画控制 | 提供关闭动画选项，尊重系统减少动画设置 |

#### 8.6.2 听觉无障碍

| 项目 | 要求 |
|------|------|
| 语音替代 | 所有语音提示同时提供文字和视觉提示 |
| 字幕支持 | 视频内容提供字幕 |
| 振动反馈 | 重要提醒同时提供振动反馈 |
| 音量控制 | 应用内语音播报音量可独立调节 |

**语音播报降级方案（三级降级策略）：**

| 优先级 | 方案 | 说明 | 触发条件 |
|--------|------|------|----------|
| 1（首选） | 预录制音频 | 辅导员创建沟通板时，系统自动将常用语预录为音频文件并下发到员工端 | 默认使用 |
| 2（补充） | 系统 TTS | 使用设备系统自带的TTS引擎进行语音合成 | 预录制音频文件缺失或损坏时 |
| 3（兜底） | 视觉展示 | 放大文字 + 闪烁边框动画，吸引对方注意 | TTS引擎不可用或被禁用时 |

**实现要求：**
- 员工端在联网时自动下载预录制音频文件并缓存
- 每次播放前检测音频文件可用性，自动选择最高优先级的可用方案
- 降级过程对用户透明，无需手动切换

#### 8.6.3 认知无障碍

| 项目 | 要求 |
|------|------|
| 界面简洁 | 每个页面只展示一个主要任务，避免信息过载 |
| 操作步骤少 | 核心操作路径不超过 3 步 |
| 语言简洁 | 使用简单、直接的语句，避免专业术语 |
| 一致性 | 相同功能使用相同的交互方式和位置 |
| 错误容忍 | 提供撤销操作，错误提示友好且明确 |
| 确认机制 | 不可逆操作（如删除）需二次确认 |
| 超时处理 | 操作超时给予充分提示，不自动跳转 |
| 帮助引导 | 提供上下文帮助信息和操作提示 |

#### 8.6.4 运动无障碍

| 项目 | 要求 |
|------|------|
| 按钮尺寸 | 所有可点击元素最小 48x48dp（员工端 64x64dp） |
| 点击间距 | 可点击元素之间保持足够间距（至少 8dp） |
| 手势替代 | 所有滑动手势同时提供按钮替代操作 |
| 长按替代 | 避免使用长按操作，改用明确的按钮 |

#### 8.6.5 屏幕阅读器支持

| 平台 | 技术 | 要求 |
|------|------|------|
| iOS | VoiceOver | 所有元素提供语义化标签（Semantics） |
| Android | TalkBack | 所有元素提供 contentDescription |
| Web | 屏幕阅读器 | 语义化 HTML + ARIA 属性 |

**Flutter 语义化标注示例：**

```dart
Semantics(
  label: '完成步骤：洗手',
  button: true,
  enabled: true,
  child: ElevatedButton(
    onPressed: () => completeStep(),
    child: Text('完成'),
  ),
)
```
