## 1 项目概述

### 1.1 项目背景

心智障碍人士（包括自闭症谱系障碍、智力障碍、唐氏综合征等群体）在融入职场过程中面临诸多挑战，如：难以理解复杂工作任务、沟通表达受限、环境适应能力较弱、情绪管理困难等。这些问题导致该群体就业率长期偏低，且在职稳定性不足。

"工作导航"项目旨在通过信息化手段，为心智障碍人士提供结构化、可视化、即时化的工作支持，帮助其更好地完成工作任务、与同事沟通协作、适应工作环境，从而提升就业质量与职场融入度。

### 1.2 项目目标

1. **短期目标**：完成产品 MVP 版本开发，覆盖核心功能模块，在试点机构进行验证。
2. **中期目标**：根据试点反馈持续迭代优化，扩展至更多就业支持机构。
3. **长期目标**：建立心智障碍人士就业支持数字化平台，形成可复制的服务模式。

### 1.3 产品定位与用户画像

**产品组成**：

| 端 | 平台 | 目标用户 | 核心价值 |
|----|------|----------|----------|
| 员工端 | Flutter 移动 APP（iOS + Android） | 心智障碍员工 | 任务引导、沟通辅助、情绪管理、环境适应 |
| 员工端（Web） | Flutter Web（同源部署于后端 `/employee-app/` 路径下） | 心智障碍员工 | 与移动端功能一致，Web 平台限制：本地通知不可用、SQLite 不可用（降级为 localStorage） |
| 辅导员端 | Web 管理后台 | 就业辅导员/管理者 | 任务管理、员工管理、数据监控、消息通知 |

**用户画像**：

- **心智障碍员工**：认知能力有限，需要简单直观的界面和操作方式；对视觉化信息接受度高于文字信息；需要即时、明确的行为引导。
- **就业辅导员**：需要高效管理多名员工的工作任务；需要实时了解员工状态；需要数据支撑来评估员工工作表现和调整支持策略。

### 1.4 核心功能模块

#### 模块一：任务拆解与视觉化流程导航（员工端 + 辅导员端）

| 功能点 | 描述 |
|--------|------|
| 任务创建与拆解 | 辅导员将工作任务拆解为步骤级子任务，支持添加图片、语音、视频指引 |
| 视觉化流程展示 | 员工端以图文卡片、流程图形式展示任务步骤，支持进度追踪 |
| 步骤引导与确认 | 逐步引导员工完成任务，每步完成后需确认，支持语音播报 |
| 任务调度与分配 | 辅导员可为不同员工分配不同任务，支持定时提醒 |
| 离线任务执行 | 支持无网络环境下加载已缓存任务并执行，联网后自动同步 |

#### 模块二：沟通板与情绪温度计（员工端 + 辅导员端）

| 功能点 | 描述 |
|--------|------|
| 图文沟通板 | 预设常用沟通卡片（如"需要帮助""休息一下"等），员工点击即可发送 |
| 自定义沟通卡片 | 辅导员可为员工定制个性化沟通卡片 |
| 情绪温度计 | 员工通过简单界面选择当前情绪状态（如开心、平静、焦虑、难过等） |
| 情绪记录与分析 | 辅导员端可查看员工情绪变化趋势，异常情绪自动预警 |
| 情绪应对建议 | 根据情绪状态推送对应的安抚策略或引导员工使用放松工具 |

#### 模块三：环境适应与即时支持工具包（员工端 + 辅导员端）

| 功能点 | 描述 |
|--------|------|
| 环境适应指南 | 针对不同工作场景（如办公室、工厂、超市等）提供环境适应指引 |
| 即时求助按钮 | 一键呼叫辅导员，支持发送位置和当前任务信息 |
| 定时提醒 | 自定义提醒事项（如喝水、休息、换岗等），支持多种提醒方式 |
| 白噪音/放松工具 | 内置白噪音播放、呼吸引导等放松功能 |
| 紧急联系人 | 快速拨打预设紧急联系人电话 |

#### 模块四：后台管理系统（辅导员端 Web）

| 功能点 | 描述 |
|--------|------|
| 员工管理 | 员工信息录入、能力评估记录、分组管理 |
| 任务管理 | 任务模板管理、任务分配、执行进度监控 |
| 消息通知 | 向员工推送任务提醒、关怀消息、公告通知 |
| 数据统计 | 员工任务完成率、情绪趋势、工作时长等数据可视化 |
| 系统设置 | 角色权限管理、系统参数配置 |

### 1.5 技术架构

#### 技术栈选型

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 移动端（员工端） | Flutter 3.x | 跨平台开发，一套代码支持 iOS、Android 和 Flutter Web |
| Web 端（辅导员端） | Vue 3 + Element Plus | 渐进式框架，适合管理后台开发 |
| 后端服务 | Node.js + Express | 高并发处理，适合实时通信场景 |
| 数据库 | MySQL 8.0 | 关系型数据库，存储结构化业务数据 |
| 缓存 | Redis | 会话管理、数据缓存、消息队列 |
| 实时消息推送 | WebSocket + Socket.IO | 即时消息与通知推送 |
| 文件存储 | 云对象存储（OSS） | 图片、音频、视频等媒体文件存储 |
| 离线同步 | SQLite（本地）+ 增量同步机制 | 移动端离线数据缓存与同步 |
| 部署环境 | 国内云服务（阿里云/腾讯云） | 满足国内合规要求，保障访问速度 |

## 2 开发团队配置建议

### 2.1 团队组织架构

| 角色 | 人数 | 职责概述 |
|------|------|----------|
| 项目经理 | 1 人 | 项目整体规划、进度管控、资源协调、风险管理、干系人沟通 |
| 产品经理 | 1 人 | 需求分析、产品原型设计、需求文档编写、用户调研、验收标准制定 |

| 前端开发（Flutter） | 2 人 | 员工端移动 APP 开发、离线功能实现、与后端接口对接 |
| 前端开发（Web） | 1 人 | 辅导员端 Web 管理后台开发、数据可视化、与后端接口对接 |
| 后端开发 | 2 人 | API 接口开发、数据库设计、实时消息推送、离线同步机制 |

**团队总规模：9 人**

### 2.2 角色与职责说明

#### 项目经理（1人）

- 制定项目计划，分解里程碑和交付节点
- 组织每日站会、周会、阶段评审会
- 跟踪项目进度，识别和解决阻塞问题
- 管理项目风险，制定应对策略
- 协调跨团队资源，对外汇报项目状态

#### 产品经理（1人）

- 深入了解心智障碍群体需求，与特教专家、就业辅导员协作
- 编写产品需求文档（PRD）和用户故事
- 设计产品原型（低保真/高保真）
- 制定功能优先级和迭代计划
- 参与用户验收测试，收集反馈并规划迭代

#### UI/UX 设计师（1人）

- 制定无障碍设计规范（大字体、高对比度、简洁布局）
- 设计员工端和辅导员端的交互流程与界面
- 制作设计稿、切图、图标资源
- 特别关注心智障碍用户的使用体验（操作简单、反馈明确、容错性强）
- 配合前端开发完成设计还原

#### 前端开发 - Flutter（2人）

- 搭建 Flutter 项目工程框架
- 实现员工端全部功能页面和交互逻辑
- 实现离线数据缓存与同步机制
- 集成推送通知、语音播报等原生能力
- 适配 iOS 和 Android 双平台

#### 前端开发 - Web（1人）

- 搭建 Vue 3 项目工程框架
- 实现辅导员端管理后台全部功能页面
- 实现数据统计图表和报表展示
- 对接后端 API 接口

#### 后端开发（2人）

- 设计数据库表结构和接口规范
- 开发 RESTful API 接口
- 实现 WebSocket 实时消息推送
- 实现离线数据增量同步机制
- 文件上传/下载服务
- 部署与运维支持

#### 测试工程师（1人）

- 编写测试计划和测试用例
- 执行功能测试、接口测试、性能测试
- 管理缺陷跟踪和回归测试
- 组织用户验收测试（UAT）
- 兼容性测试（多机型、多浏览器）

### 2.3 团队协作模式

- **敏捷开发模式**：采用 Scrum 框架，以 2 周为一个 Sprint 周期
- **每日站会**：每天 15 分钟，同步昨日进展、今日计划、遇到的障碍
- **代码管理**：Git + GitFlow 分支管理策略
- **需求管理**：使用看板工具（如飞书项目/Teambition）进行任务跟踪
- **文档管理**：使用在线文档协作平台，保持文档实时更新
- **沟通渠道**：即时通讯工具 + 定期会议

## 7 质量保障计划

### 7.1 代码规范

#### Flutter（Dart）代码规范

| 规范项 | 具体要求 |
|--------|----------|
| 代码风格 | 遵循 [Effective Dart](https://dart.dev/guides/language/effective-dart) 规范 |
| 命名规范 | 类名使用 UpperCamelCase，变量/方法名使用 lowerCamelCase，常量使用 lowerCamelCase |
| 文件组织 | 按功能模块划分目录（lib/features/模块名/），每个模块包含 models、views、widgets、controllers |
| 注释规范 | 公共 API 必须添加文档注释，复杂逻辑添加行内注释 |
| 状态管理 | 统一使用 Provider 6.x 进行状态管理，禁止混用多种方案 |
| 代码格式化 | 使用 `dart format` 统一格式化，IDE 配置保存时自动格式化 |
| 静态分析 | 使用 `dart analyze` 进行静态检查，零警告零错误方可合并 |

#### Vue（JavaScript）代码规范

| 规范项 | 具体要求 |
|--------|----------|
| 代码风格 | 遵循 ESLint + Prettier 配置 |
| 命名规范 | 组件名使用 PascalCase，文件名使用 kebab-case |
| 组件规范 | 单文件组件（SFC）结构顺序：`<template>` → `<script>` → `<style>` |
| TypeScript | 使用 JavaScript (ES Module)，接口定义与后端 API 保持一致 |
| API 管理 | 统一在 api 目录下按模块管理，使用 Axios 实例封装 |
| 代码格式化 | 使用 Prettier 统一格式化，Husky + lint-staged 保证提交代码质量 |

#### Node.js（JavaScript）代码规范

| 规范项 | 具体要求 |
|--------|----------|
| 代码风格 | 遵循 ESLint + Prettier 配置（Airbnb 风格） |
| 项目结构 | 按功能模块划分目录（routes/、controllers/、services/、models/、middlewares/） |
| 错误处理 | 统一错误处理中间件，自定义业务异常类 |
| 日志规范 | 使用 Winston 日志库，区分日志级别（error、warn、info、debug） |
| 接口规范 | RESTful 风格，统一响应格式 `{ code, message, data }` |
| 数据库操作 | 使用原生 SQL（mysql2 驱动），参数化查询防止注入 |

### 7.2 代码审查

#### 审查流程

```
开发者提交代码 → 创建 Merge Request / Pull Request
        ↓
自动化检查（Lint、格式化、单元测试）
        ↓
至少 1 名同行审查（Peer Review）
        ↓
审查通过 → 合并到主分支
审查不通过 → 修改后重新提交
```

#### 审查要求

| 要求项 | 具体说明 |
|--------|----------|
| 审查范围 | 所有代码变更必须经过 Code Review，包括功能代码、配置文件、文档 |
| 审查人数 | 每个 MR/PR 至少需要 1 名同行审查通过方可合并 |
| 审查时效 | 提交审查后 24 小时内完成审查 |
| 审查重点 | 1. 代码逻辑正确性；2. 代码规范遵循情况；3. 潜在性能问题；4. 安全漏洞；5. 可维护性 |
| 审查工具 | 使用 GitLab/GitHub 内置的 Merge Request / Pull Request 功能 |
| 审查记录 | 审查意见记录在 MR/PR 评论中，重大问题需创建 Issue 跟踪 |

| 3. 员工端测试 | 心智障碍员工在辅导员引导下完成指定任务操作 | 产品经理、辅导员 | 2 天 |
| 4. 辅导员端测试 | 辅导员独立完成管理后台各项操作 | 产品经理 | 1 天 |
| 5. 访谈与反馈 | 收集测试人员的主观感受和改进建议 | 产品经理 | 1 天 |
| 6. 结果整理 | 汇总测试结果，分类整理问题，确定优先级 | 产品经理、测试工程师 | 1 天 |

#### UAT 评估标准

| 评估维度 | 通过标准 |
|----------|----------|
| 任务完成率 | 心智障碍员工能够独立完成（或在辅导员口头提示下完成）80% 以上的核心任务 |
| 操作错误率 | 核心操作错误率低于 20%。具体定义：在员工端执行完整任务流程（从打开任务到完成所有步骤）的过程中，误触非目标按钮（如误点返回、误点其他任务卡片等）导致流程中断的次数，占总操作次数的比例不超过 20%。统计方式：测试记录员记录每位测试员工的总点击次数和误触次数，计算误触率。示例：员工完成一个5步任务共点击12次，其中误触2次，误触率为 16.7%，低于 20% 阈值，判定通过。 |
| 满意度 | 辅导员对管理功能的满意度评分 >= 4/5 |
| 严重问题数 | 无致命级（P0）和严重级（P1）问题 |

### 7.6 持续集成与持续部署

#### CI/CD 流程

```
代码提交 → 自动化构建 → 单元测试 → 代码质量检查
    ↓
构建成功 → 自动部署到测试环境 → 接口自动化测试
    ↓
测试通过 → 人工验证 → 部署到预发布环境 → 验收测试
    ↓
验收通过 → 部署到生产环境
```

#### CI/CD 工具链

| 环节 | 工具 | 说明 |
|------|------|------|
| 代码托管 | GitLab / GitHub | 代码版本管理 |
| CI 流水线 | GitLab CI / GitHub Actions | 自动化构建、测试 |
| 代码质量 | SonarQube | 静态代码分析、代码质量度量 |
| 制品管理 | Docker + Harbor | 容器化构建与镜像管理 |
| 部署工具 | Docker Compose / K8s | 环境部署 |
| 监控告警 | Prometheus + Grafana | 服务监控与告警 |

#### 环境规划

| 环境 | 用途 | 数据 | 更新频率 |
|------|------|------|----------|
| 开发环境（Dev） | 日常开发调试 | 模拟数据 | 实时 |
| 测试环境（QA） | 功能测试、自动化测试 | 测试数据 | 每日构建 |
| 预发布环境（Staging） | 验收测试、上线前验证 | 生产数据脱敏 | 发版前 |
| 生产环境（Production） | 正式对外服务 | 真实数据 | 发版时 |
